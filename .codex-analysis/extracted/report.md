## 3. Understanding the Jio Hotstar Landscape

### 3.1. Platform Architecture: What Powers the Streaming

Jio Hotstar operates one of the most technically demanding streaming infrastructures in the world, architected to handle the massive concurrency spikes characteristic of live cricket broadcasting in India. The platform's backend was fundamentally redesigned under an internal initiative known as "Project Hulk," which transitioned the system from a monolithic architecture to a microservices-oriented paradigm deployed across extensive Kubernetes clusters on Amazon Web Services (AWS). The infrastructure utilizes over one hundred thousand computing cores and hundreds of terabytes of RAM to ensure seamless delivery during peak events. The networking layer is optimized to prevent NAT gateway and VPC configuration bottlenecks that commonly afflict large-scale cloud deployments, using continuous benchmarking, VPC flow log analysis, and stepped node provisioning to enable automatic scaling without human intervention. This architectural context is critical for understanding why Jio Hotstar does not expose public APIs — the system's primary design objective is resilience and scale, not third-party extensibility. Every external-facing endpoint represents a potential attack vector and scaling liability, which explains the platform's closed ecosystem approach.

The platform's content delivery infrastructure is equally sophisticated. Video delivery employs Adaptive Bitrate Streaming (ABR) via HTTP Live Streaming (HLS) and MPEG-DASH protocols, dynamically adjusting quality based on the user's available bandwidth. A multi-CDN strategy incorporating Akamai, Cloudflare, and AWS CloudFront optimizes global delivery, with an origin shield protecting core packaging and storage servers from request overload. Content security is enforced through multi-DRM solutions: Google Widevine for Android and web browsers, Apple FairPlay for iOS and Apple TV, and Microsoft PlayReady for Windows platforms. Access to video streams requires short-lived, cryptographically signed edge tokens. This robust security framework makes any attempt to intercept, proxy, or decrypt video streams technically infeasible and legally hazardous. The architectural implication is clear: any custom recommendation layer must operate exclusively on metadata and must delegate playback entirely to Jio Hotstar's secure native environment.

The Backend-for-Frontend (BFF) architectural pattern is particularly relevant to the browser extension approach. Jio Hotstar employs BFF to minimize latency: instead of client applications making multiple chatty requests to disparate microservices (authentication, personalization, content metadata), the BFF layer intercepts client requests, parallelizes internal calls, aggregates the data, and returns a single optimized payload tailored to the requesting device's capabilities. This layer also implements multi-tiered caching using Redis clusters for high-frequency queries and in-process caches for critical keys. Understanding the BFF pattern explains why the Jio Hotstar website loads content dynamically as the user scrolls — the frontend is essentially a shell that renders whatever the BFF sends. The browser extension's MutationObserver must detect these dynamically injected content cards as they arrive from the BFF, process them against the user's blocklist, and either hide them or enrich them with additional metadata and swipe controls.

### 3.2. The No-Public-API Reality

Jio Hotstar does not publish a public developer API for content catalog, user data, or recommendations. All data APIs are internal to their applications, accessed via authenticated sessions with platform-specific headers. While community developers have occasionally reverse-engineered private endpoints — for example, a Kodi plugin calls an internal JSON endpoint at `/o/v2/menu` with Android platform headers to fetch menu items — these approaches require valid Jio Hotstar authentication tokens and are explicitly against the platform's Terms of Service. Attempting to use internal APIs risks account bans and potential legal action under anti-scraping and computer fraud statutes. This constraint is non-negotiable and fundamentally shapes the architecture: the recommendation overlay cannot rely on any direct API integration with Jio Hotstar. Instead, it must source content metadata from third-party databases (TMDb, OMDb), verify availability through independent aggregators (JustWatch, 91mobiles), and interact with the Jio Hotstar platform only through standard web browsing (for the extension) or deep-link URL opening (for playback). This constraint is actually liberating in some respects — it forces the architecture to be completely decoupled from Jio Hotstar's internal systems, making it more resilient to platform changes and legally unassailable.

The absence of a public API is not unusual in the streaming industry. Netflix, Amazon Prime Video, and Disney+ (globally) similarly do not offer public APIs for their content catalogs. This pattern reflects the intensely competitive nature of content licensing, where exclusive distribution rights are fiercely protected. A public API would enable competitors to monitor content additions and removals in real-time, potentially undermining negotiated exclusivity windows. It would also create a support burden that platforms are unwilling to bear. For the user's project, this constraint simply means accepting that Jio Hotstar's catalog data must be sourced indirectly and that the system will always have a slight lag (hours to days) in reflecting the platform's current content library. The 91mobiles database, which tracks 4,317 movies and 1,682 TV shows on Jio Hotstar, provides a reasonable approximation of the catalog, supplemented by TMDb's watch providers endpoint which can verify whether a specific title is currently available on the platform in India.

### 3.3. Content Discovery: How Recommendations Currently Work

Jio Hotstar's native recommendation engine operates through a sophisticated real-time data pipeline. User interactions — playbacks, pauses, searches, drop-offs, completion rates — are ingested as telemetry beacons via an Apache Kafka event bus. Apache Flink processes this raw behavioral stream in real-time, feeding analyzed signals into an Online Analytical Processing (OLAP) data warehouse and a centralized feature store. This architecture allows the platform's machine learning models to adjust rankings and generate dynamic recommendations based on immediate user context. The underlying algorithmic framework typically combines several methodologies: collaborative filtering identifies patterns across users with similar viewing histories; content-based filtering uses metadata (genres, cast, directors) to match items; non-negative matrix factorization (NMF) decomposes user-item interaction matrices into latent feature spaces; and hybrid systems combine these approaches, often using deep neural networks to blend signals. Despite this sophistication, these algorithms universally suffer from a lack of deterministic user control. They optimize for aggregate engagement metrics — total watch time, click-through rates, retention — rather than individual user satisfaction. If the model infers that a user enjoys political thrillers based on one watched title, it will continuously surface that genre without any mechanism for the user to explicitly reject specific titles or indicate that their interest was situational rather than preference-based.

The recommendation pipeline typically follows a multi-stage architecture: candidate generation (selecting thousands of potentially relevant items from the catalog), scoring (ranking candidates using the ML model), and re-ranking (applying business rules, diversity constraints, and freshness heuristics). The user's frustration arises primarily in the re-ranking stage, where business objectives (promoting original content, fulfilling licensing commitments, maximizing ad inventory) often override user preferences. The overlay system proposed in this report essentially replaces Jio Hotstar's re-ranking stage with a user-controlled alternative, while still operating on the same candidate pool. This is a powerful conceptual framework: the overlay doesn't need to generate candidates (Jio Hotstar's existing discovery mechanisms — search, genre browsing, trending — already do this), it needs to intelligently filter and rank them according to the user's explicit preferences.

### 3.4. Existing Precedents: Sift, Rating-Finder, and Similar Tools

The feasibility of browser extension-based content filtering on Jio Hotstar is not merely theoretical — it has already been demonstrated by existing tools. **Sift** is a browser extension explicitly designed to hide low-rated shows on Jio Hotstar, automatically filtering out content below a user-configurable IMDb threshold. It operates by scanning the Jio Hotstar DOM, extracting show titles, querying IMDb for ratings, and applying `display:none` to items that fall below the threshold. **Rating-Finder** is another Chrome extension that reads the Jio Hotstar page and overlays IMDb ratings onto each movie tile, enriching the browsing experience with external quality signals. These extensions prove that DOM manipulation on the Jio Hotstar website is technically viable and that the platform has not aggressively blocked such modifications. The legal precedent for user-controlled browser modifications is further supported by the broader ecosystem of content filtering extensions: **Unhook** for YouTube (removes recommended videos, comments, and trending sections), **AdBlock Plus** (filters advertising content), and **Stylus** (applies custom CSS to any website) all operate on the same fundamental principle — the user has the right to modify how content is rendered in their own browser. No court has ruled against personal-use browser extensions that modify DOM presentation, and the established legal interpretation is that once content is delivered to the user's device, the user may control how it is displayed, provided they do not redistribute modified content or circumvent access controls.

These precedents provide both technical validation and legal reassurance. Sift's existence demonstrates that title extraction from the Jio Hotstar DOM is possible (even if the specific selectors it uses are not publicly documented). Rating-Finder shows that IMDb data can be fetched and overlaid in real-time. Unhook proves that aggressive content filtering — removing entire sections of a platform's UI — can be done without legal consequence. The key distinction that keeps all these tools legally safe is that they operate on the client side, after content has been legitimately delivered to an authenticated user. They do not bypass authentication, do not scrape backend APIs, do not redistribute content, and do not interfere with other users' experiences. The overlay system proposed in this report follows these same principles precisely.

## 4. The Synthesized Architecture: A Dual-Client Approach

### 4.1. Client Layer: Two Entry Points, One Unified Backend

The synthesized architecture employs two complementary client interfaces that share a single backend, catalog, and user profile. This dual-client approach ensures coverage across all user scenarios: real-time filtering while browsing Jio Hotstar, and immersive discovery through a dedicated interface.

#### 4.1.1. Browser Extension (Chrome Manifest V3): Real-Time DOM Filtering

The Chrome Extension is the primary tool for real-time content filtering on the Jio Hotstar website (`hotstar.com`). It operates as a Manifest V3 extension with three core components: a **content script** that runs in the context of the Jio Hotstar page, a **background service worker** that handles API calls and data persistence, and a **popup UI** for quick settings access. The content script uses `MutationObserver` to detect when new content cards are dynamically injected into the DOM by Jio Hotstar's React frontend. When a mutation is detected, the script extracts the title from the card (using `alt` text on images, URL patterns like `/movies/` or `/shows/`, or `data-testid` attributes if present), checks it against the user's local blocklist (stored in `chrome.storage.local` and cached in memory for sub-millisecond lookups), and immediately applies `display:none` to matching cards. For cards that are not blocked, the script injects a **Shadow DOM overlay** containing swipe controls (left=block, right=like, up=watched, down=watch-later), the IMDb rating fetched from local cache or OMDb API, and genre tags. The Shadow DOM encapsulation ensures that the extension's CSS does not conflict with Jio Hotstar's styling. The background service worker handles all external API calls (to TMDb, OMDb, and the optional backend), manages IndexedDB for the local catalog cache, and syncs user interactions to the backend when cloud sync is enabled. A debounced processing pipeline ensures that rapid DOM mutations (during infinite scroll) are batched and processed efficiently using `requestIdleCallback` for non-critical enrichment tasks.

The extension's technical implementation must address several challenges specific to Jio Hotstar's React SPA architecture. First, the initial page load does not contain all content — carousels are loaded dynamically as the user scrolls, and some content is loaded via XHR/fetch requests that modify the DOM. The MutationObserver must be configured with `{ childList: true, subtree: true }` to catch all these changes, but this can generate a high volume of mutation records. A debounced callback that batches mutations and processes them on `requestAnimationFrame` (for blocking) and `requestIdleCallback` (for enrichment) ensures smooth performance. Second, Jio Hotstar uses obfuscated CSS class names (e.g., `css-1a2b3c4`) that change between deployments. The extension must rely on stable selectors: semantic HTML elements (`article`, `figure`), `data-testid` attributes (if Jio Hotstar uses them consistently), URL patterns in anchor tags, and image `alt` text. A robust fallback strategy tries multiple selector patterns and gracefully degrades if none match. Third, the extension must detect and handle different page types — homepage, search results, genre pages, and the "Watch" page — each with slightly different DOM structures. A page-type detection function that examines the URL path and DOM structure at load time configures the appropriate selectors and processing logic.

#### 4.1.2. Standalone PWA (React + Vite): The Discovery Portal

The Standalone Progressive Web App serves as the dedicated discovery and curation interface, optimized for the immersive swipe experience modeled after Google Movies. Built with React and Vite for fast loading and deployed as a static site on Vercel or Netlify, the PWA can be "Added to Home Screen" on mobile devices, behaving like a native app. The PWA offers two primary views: a **Swipe Deck** (full-screen, Tinder-style card stack powered by `react-tinder-card` with Framer Motion physics) for rapid binary feedback, and a **Discovery Grid** (responsive masonry layout with filter sidebar) for browsing by genre, year, rating, or mood. Both views operate on the same catalog and recommendation engine but serve different user intents — the Swipe Deck is for quick, instinctive curation when the user wants to train the AI rapidly, while the Discovery Grid is for intentional exploration when the user has a specific mood or criteria in mind. Every content card displays rich metadata from TMDb/OMDb: poster, title, IMDb rating, genres, runtime, cast, plot summary, and a "Why this?" button that reveals the AI's explanation for the recommendation. Clicking "Watch" constructs the appropriate Jio Hotstar deep link and opens it in a new tab or mobile app. The PWA uses service workers for offline catalog access (cached enriched metadata) and background sync to queue interactions when connectivity is intermittent. Since it operates independently of the Jio Hotstar website, the PWA is immune to DOM changes on the platform and provides a stable, controllable user experience.

The PWA's design prioritizes the swipe interaction as the primary mode of engagement. The swipe deck presents one full-screen card at a time, showing the movie poster as a background with gradient overlay, title, rating, and a one-line plot teaser. Swipe left triggers a satisfying animation (card flies off-screen left, red "BLOCKED" stamp appears) and permanently adds the item to the blocklist. Swipe right triggers a green "LIKED" animation and adds the item to the positive training set. Swipe up marks as watched with a blue checkmark. Swipe down adds to watch-later with a yellow bookmark. Each swipe updates the taste vector and triggers a lightweight re-ranking of the remaining candidate pool. The physics of the swipe — spring tension, velocity sensitivity, rotation during drag — are tuned to feel responsive and satisfying, encouraging continued engagement. After every 10 swipes, the system surfaces a "You have [N] new recommendations based on your swipes!" message, providing positive reinforcement for the curation activity and making the learning loop tangible.

#### 4.1.3. Unified User Profile: Shared Blocklist, History, and Taste Vector

Both clients share a unified user profile that persists across devices when cloud sync is enabled. The profile consists of four primary data structures: the **Blocklist** (a set of content IDs that are permanently excluded from all recommendations, with timestamps and optional user-provided reasons), the **Watch History** (timestamped records of completed or partially watched content, used for re-recommendation prevention), the **Taste Vector** (a 384-dimensional floating-point array representing the user's aggregated preference embedding, updated after every swipe), and the **Interaction Log** (a chronological record of all swipes, blocks, likes, and watches, used for model training and the agentic AI's context window). Local-first storage uses `chrome.storage.local` (10MB+ with `unlimitedStorage` permission) for the blocklist and settings, and IndexedDB via Dexie.js for the catalog, embeddings, and interaction history. When cloud sync is enabled, the background service worker syncs these data structures to the backend API using delta updates (only changed records) to minimize bandwidth. Conflict resolution uses last-write-wins with optional manual merge for edge cases. The shared profile ensures that a movie blocked on the Jio Hotstar website via the extension never appears in the PWA's swipe deck, and a movie liked in the PWA immediately influences the taste vector used for filtering on the website.

The data model for the user profile is designed for efficiency and privacy. The blocklist is stored as a hash set (JavaScript `Set` or object map) to enable O(1) membership testing — critical for real-time filtering where hundreds of cards may need to be checked per page load. The watch history stores content IDs with timestamps and completion percentages (100% for fully watched, partial for abandoned), allowing the system to distinguish between completed and sampled content. The taste vector is the most frequently updated field: after every swipe, the embedding of the swiped item is either added (like) or subtracted (dislike) from the running average, with a learning rate decay that ensures early swipes have less influence than recent ones. This exponential moving average approach prevents the taste vector from becoming dominated by initial preferences and allows it to evolve as the user's tastes change. The interaction log serves as the raw training data for the Two-Tower Neural Network and the context window for the Agentic AI Reranker, providing a complete history of explicit feedback that these systems can learn from.

### 4.2. Service Layer: The Backend That Binds Everything

The service layer provides the shared infrastructure that both clients depend on, implemented as lightweight, serverless-friendly functions that can run on free tiers for personal use.

#### 4.2.1. Catalog Sync Service: Keeping Content Fresh

The Catalog Sync Service maintains an up-to-date, enriched database of movies and shows available on Jio Hotstar. It operates as a scheduled pipeline (weekly GitHub Actions cron job or a Vercel cron function) with three stages. **Stage 1 — Discovery**: A Python scraper using BeautifulSoup or Scrapy navigates the 91mobiles Jio Hotstar movie list, extracting titles, Jio Hotstar URLs, age ratings, genres, and cast information. The 91mobiles page currently lists 4,317 movies and 1,682 TV shows, providing a comprehensive baseline catalog. **Stage 2 — Enrichment**: For each discovered title, the pipeline queries the TMDb API search endpoint (`/search/multi`) to obtain the TMDb ID, full metadata (genres as structured IDs, cast, crew, overview, poster path, backdrop, release date, runtime), and watch provider information (`/movie/{id}/watch/providers` with `watch_region=IN` to verify Jio Hotstar availability). TMDb's free tier permits 40 requests per 10 seconds, which is sufficient for incrementally updating a few hundred changed titles per week. For IMDb-specific data (ratings, parental guidance), the pipeline queries OMDb API (1,000 free requests per day, adequate for weekly batch enrichment). **Stage 3 — Publication**: The enriched catalog is serialized as a versioned JSON file (`enriched_catalog_v{N}.json`) and published to a static CDN (Cloudflare R2 free tier, or simply committed to the GitHub repository). Both clients download this catalog on first load and check for updates periodically (daily for the PWA, weekly for the extension). The catalog schema includes fields for `id`, `title`, `original_title`, `tmdb_id`, `imdb_id`, `jiohotstar_url`, `genres` (array of IDs), `genre_names`, `overview`, `cast`, `director`, `poster_url`, `backdrop_url`, `imdb_rating`, `tmdb_rating`, `runtime`, `release_date`, `age_rating`, `embedding_384` (precomputed), and `last_updated`. Precomputing embeddings during the sync pipeline eliminates the need for client-side embedding generation, reducing runtime dependencies.

The catalog sync pipeline is designed for reliability and incremental efficiency. Rather than re-scraping the entire catalog every week, the pipeline maintains a checkpoint of previously seen titles and only processes new or changed entries. A title is considered "changed" if its 91mobiles page has been modified since the last sync (detected via HTTP `Last-Modified` header or ETag), if TMDb has updated its metadata (detected via TMDb's `last_modified` field), or if OMDb has a new IMDb rating. This incremental approach typically results in only 50-200 titles requiring enrichment per week, well within API rate limits. The pipeline also implements exponential backoff for API failures, alert notifications (via email or webhook) for consecutive failures, and automatic retry with circuit breaker pattern to handle temporary API outages gracefully.

#### 4.2.2. Preference API: User Data, Blocklist, and History Management

The Preference API is a lightweight REST API (FastAPI for Python or Express for Node.js) deployed on a free tier (Railway, Render, or Fly.io) that handles user data synchronization and advanced operations. For users who do not enable cloud sync, this API is optional — all data remains local. For synced users, it provides endpoints for: `GET /profile` (retrieve the user's complete profile including taste vector), `POST /block` (add an item to the blocklist with optional reason), `DELETE /unblock` (remove from blocklist), `GET /blocklist` (retrieve full blocklist for client cache refresh), `POST /interact` (log a swipe interaction with type, content ID, timestamp, and context), `GET /recommendations` (retrieve AI-ranked candidates, accepting parameters for count, genre filter, mood, and exploration ratio), and `POST /watch` (mark an item as watched). The API implements rate limiting (100 requests per hour per device) and uses JWT tokens signed with a device-specific key for lightweight authentication without requiring user accounts. The recommendation endpoint is the most complex: it retrieves candidate items from the catalog, applies hard filters (blocklist exclusion, watched exclusion, recently shown cooldown), scores remaining items using the user's taste vector (cosine similarity against precomputed embeddings), injects exploration items via epsilon-greedy selection, and returns the ranked list with explanation metadata. For users who have enabled the agentic AI layer, this endpoint also triggers the LangGraph pipeline for LLM-based reranking and explanation generation.

The API design prioritizes low latency for the critical path. The `/recommendations` endpoint is optimized for sub-100ms response times through aggressive caching: the user's blocklist is cached in Redis as a SET data structure enabling O(1) exclusion checks, the catalog embeddings are cached in memory on the API server, and recommendation results are cached for 5 minutes to prevent redundant computation. The `/interact` endpoint is designed for fire-and-forget semantics: it acknowledges the interaction immediately, queues the model update for asynchronous processing, and returns a 202 Accepted response. This ensures that swipe actions feel instantaneous while allowing the taste vector update and neural network retraining to occur in the background. The API also supports batch operations (`POST /interact/batch`) for syncing offline interactions when the client reconnects, ensuring that PWA usage without connectivity does not lose training data.

#### 4.2.3. Deep Link Engine: Seamless Playback Handoff

The Deep Link Engine ensures that when a user clicks "Watch" on any recommended item, they are taken directly to the correct playback location in Jio Hotstar's secure environment without any friction. The engine maintains a mapping between content IDs and Jio Hotstar URLs, extracted during the catalog sync process. For **web playback**, it constructs the standard Jio Hotstar URL (`https://www.hotstar.com/in/movies/{movie_title}/{internal_id}/watch` for movies, `https://www.hotstar.com/in/shows/{show_title}/{internal_id}` for shows) and opens it in a new tab. If the user has an active authenticated session, playback begins immediately; if not, Jio Hotstar's native login flow takes over. For **mobile app launching**, the engine generates an Android intent URL: `intent://details?id={internal_id}#Intent;scheme=hotstar;package=in.startv.hotstar;S.browser_fallback_url=https%3A%2F%2Fwww.hotstar.com%2Fin%2Fmovies%2F{title}%2F{id}%2Fwatch;end`. This intent instructs the Android system to open the Jio Hotstar app directly to the specified content, with a fallback to the web URL if the app is not installed. For **iOS**, universal links (`https://www.hotstar.com/...`) typically route to the native app automatically when installed. The engine also handles edge cases: sports content uses `/sports/` paths, live events use `/live-sports/`, and some legacy content may use different URL patterns. The critical security principle is that the engine never attempts to proxy, embed, or manipulate the video stream — it only generates navigation URLs, ensuring full DRM compliance and zero security bypass.

The Deep Link Engine also implements smart platform detection to optimize the handoff experience. When a user clicks "Watch" in the PWA on an Android device, the engine checks whether the Jio Hotstar app is installed (via the `navigator.getInstalledRelatedApps()` API, if supported) and generates the appropriate intent URL. On iOS, it uses the universal link format and relies on the operating system's automatic app routing. On desktop, it opens the web URL in a new tab with `window.open()`. The engine also tracks deep link success rates (did the user successfully start playback within 30 seconds of clicking?) to identify broken URLs or platform changes, feeding this data back to the catalog sync service for URL validation. A periodic "link health check" process visits all stored Jio Hotstar URLs to verify they still resolve to valid content pages, flagging any that return 404 or redirect to the homepage for re-scraping.

### 4.3. AI/ML Layer: The Three-Layer Recommendation Stack

The recommendation engine is organized into three progressive layers that deliver immediate value and continuously improve as user interaction data accumulates. This layered approach avoids the cold-start problem (the system works from day one using pretrained embeddings) while providing a clear path to increasing sophistication.

#### 4.3.1. Layer 1: Semantic Embeddings with Sentence Transformers

The foundation of the recommendation system is a semantic embedding of every movie and show in the catalog, generated using the `all-MiniLM-L6-v2` Sentence Transformer model. This model was selected after careful evaluation of the trade-offs between accuracy and computational efficiency. It produces 384-dimensional dense vectors, runs at approximately 14,000 sentences per second on a CPU (compared to 4,000 for the larger `all-mpnet-base-v2`), and has a small memory footprint (22 million parameters, ~80MB on disk) suitable for client-side execution. The embedding for each title is computed from a concatenated text representation: `[Title]. [Genres]. [Overview]. Starring [Top 3 Cast]. Directed by [Director].` — for example, "The Dark Knight. Action, Crime, Drama. When the menace known as the Joker wreaks havoc on the people of Gotham, Batman must accept one of the greatest psychological and physical tests of his ability to fight injustice. Starring Christian Bale, Heath Ledger, Aaron Eckhart. Directed by Christopher Nolan." These embeddings are precomputed during the catalog sync pipeline and stored alongside each catalog entry, eliminating runtime embedding generation. When the user first installs the extension or opens the PWA, the system can immediately provide content-based recommendations by computing cosine similarity between the user's initial preference profile (derived from onboarding genre selections or an initial "Preference Genome" survey) and all item embeddings. This layer requires zero training data and zero model training — it works instantly using the semantic understanding encoded in the pretrained model. As the user swipes, liked item embeddings are averaged into a running "taste vector" that represents the user's evolving preferences.

The embedding strategy captures multiple dimensions of content similarity that traditional genre-based matching misses. Two films may share no genres but have similar narrative themes, tones, or character dynamics that the transformer model encodes in the embedding space. For example, a user who likes "The Social Network" (Drama) might also appreciate "Moneyball" (Drama, Sport) or "Steve Jobs" (Drama, Biography) — the embedding captures the shared "tech industry drama" semantic even though the genres differ. The 384-dimensional space is rich enough to encode these nuanced relationships while being compact enough to store thousands of embeddings locally and compute similarities quickly. The taste vector update mechanism uses an exponential moving average with a learning rate of 0.1, meaning each new swipe contributes 10% to the updated vector while preserving 90% of the existing preference state. This ensures stability (individual swipes don't dramatically shift recommendations) while allowing gradual adaptation to evolving tastes.

#### 4.3.2. Layer 2: Two-Tower Neural Network in TensorFlow.js

Once the user has accumulated approximately 50-100 swipe interactions, the system activates the Two-Tower Neural Network — a lightweight deep learning model that learns nonlinear preference boundaries beyond what linear embedding similarity can capture. The architecture consists of two parallel neural networks: the **User Tower** takes the user's swipe history (sequence of liked and disliked item embeddings, plus metadata like swipe timing and context) and outputs a 64-dimensional user embedding. The **Item Tower** takes a candidate item's features (the 384-dim semantic embedding, normalized IMDb rating, release year, runtime, and genre one-hot encoding) and outputs a 64-dimensional item embedding. The predicted relevance score is the dot product of these two embeddings. The model is implemented in TensorFlow.js and trained entirely in the browser (or in the PWA's Web Worker to avoid blocking the UI), ensuring that raw swipe data never leaves the user's device unless cloud sync is explicitly enabled. Training uses contrastive loss: liked items are pushed closer to the user embedding, disliked items are pushed farther away. The model retrains incrementally after every 10 new swipes, with each training session taking less than 2 seconds on modern hardware. This layer captures complex, nonlinear preference patterns that semantic similarity alone cannot — for example, "I like thrillers, but only if they star certain actors and were made after 2010, and I dislike those with excessive violence." The Two-Tower architecture is the industry standard for large-scale recommendation (used by YouTube, Google Play, and Pinterest) and provides excellent performance even with limited training data.

The TensorFlow.js implementation uses a simple but effective architecture. The User Tower is a 3-layer MLP: input layer (384-dim average of liked items + 10-dim interaction statistics), hidden layer (128 units, ReLU activation), hidden layer (64 units, ReLU activation), output layer (64 units, linear activation). The Item Tower is similarly structured: input layer (384-dim embedding + 1-dim rating + 1-dim year + 1-dim runtime + 20-dim genre one-hot = 407 total), hidden layer (128 units, ReLU), hidden layer (64 units, ReLU), output layer (64 units, linear). Both towers use L2 regularization (0.001) to prevent overfitting on limited personal data. The contrastive loss function computes the difference between the dot product of the user embedding with a positive example (liked item) and the dot product with a negative example (disliked item), maximizing this margin using hinge loss with a margin of 1.0. Training uses the Adam optimizer with a learning rate of 0.001, batch size of 32, and runs for 50 epochs per retraining session. The entire model fits in under 5MB of memory and trains in a Web Worker without impacting UI responsiveness.

#### 4.3.3. Layer 3: Agentic AI Reranker with LangGraph and Local LLMs

The top layer of the recommendation stack is an Agentic AI system that provides explainable recommendations, conversational refinement, and intelligent exploration. This layer is optional (it requires running a local LLM or connecting to an API) and activates when the user requests explanations or uses the natural language search feature. The system is implemented as a **LangGraph state machine** with five nodes. **Node 1 — Context Retrieval**: The agent queries the user's profile (taste vector, recent swipes, blocklist) and the current recommendation candidate set. **Node 2 — Semantic Search**: If the user has entered a natural language query (e.g., "something like Inception but shorter"), the query is embedded using the same Sentence Transformer and used to search the vector database for semantically similar items. **Node 3 — Preference Parsing**: The LLM analyzes the user's swipe history and writes a natural-language "taste profile" (e.g., "User enjoys cerebral sci-fi and crime thrillers with twist endings, dislikes romantic comedies and horror, prefers films under 2.5 hours, has a strong preference for Christopher Nolan and Denis Villeneuve"). **Node 4 — Reranking**: The LLM receives the candidate items, the taste profile, and any natural language query, then reranks the candidates and generates a one-line explanation for each recommendation (e.g., "Recommended because you liked Inception — this is a mind-bending sci-fi thriller with a complex narrative structure and strong critical reception"). **Node 5 — Exploration Injection**: The agent identifies underexplored genres or directors that align with the user's taste profile but have few interactions, intentionally injecting diverse candidates to prevent filter bubbles. For local execution, the system uses **Ollama** running **Llama 3.2** (or a similarly lightweight model) on the user's machine, eliminating API costs and ensuring complete privacy. For users without sufficient hardware, the system can fall back to cloud LLM APIs (OpenAI, Claude) with explicit consent.

The LangGraph state machine provides deterministic control over what would otherwise be an unpredictable LLM interaction. Unlike a simple prompt-chain approach where the LLM receives a single prompt and returns a result, the graph structure allows for branching logic, error handling, and stateful memory. For example, if the Semantic Search node returns fewer than 5 candidates, the graph branches to a "Broaden Search" node that relaxes filters before proceeding to the Reranking node. If the Preference Parsing node produces an incoherent taste profile (detected by a validation heuristic), the graph loops back to retry with additional context. The state object carries forward through the graph, accumulating context at each node, and the final output includes not just the reranked recommendations but also the generated taste profile (visible to the user in a "My Taste" section) and explanations for each recommendation. This transparency builds user trust and provides debugging information if recommendations seem off.

### 4.4. Data Layer: Storage, Caching, and Vector Search

#### 4.4.1. Local-First Storage: chrome.storage, IndexedDB, and Dexie.js

The architecture prioritizes local data storage to ensure privacy and minimize backend dependencies. The **Blocklist** is stored in `chrome.storage.local` as a simple object map (`{ [contentId]: { title, reason, timestamp } }`) for O(1) lookup performance during real-time filtering. With the `unlimitedStorage` permission, there is no practical size limit. **User Settings** (exploration ratio, minimum IMDb threshold, genre preferences) use `chrome.storage.sync` (100KB limit) to automatically synchronize across devices where the user is logged into Chrome. **The Full Catalog** (enriched metadata + precomputed embeddings) is stored in IndexedDB via Dexie.js, providing a structured queryable database that can handle the full 4,300+ movie catalog (approximately 20MB with embeddings). **Interaction History** (swipe logs) is also stored in IndexedDB, partitioned by month for efficient querying. **The Taste Vector** and **Two-Tower Model Weights** are stored in `chrome.storage.local` as serialized Float32Arrays. This multi-storage strategy optimizes for the access patterns of each data type: frequent small reads (blocklist → storage.local), large structured queries (catalog → IndexedDB), and cross-device sync (settings → storage.sync). All storage operations are asynchronous (Promises) to prevent blocking the UI thread.

The IndexedDB schema is designed for efficient querying. The `catalog` object store uses `contentId` as the primary key, with indexes on `imdbRating` (for threshold filtering), `genres` (for multiEntry genre queries), `releaseYear` (for date range filtering), and `embedding` (for vector similarity search via linear scan). The `interactions` object store uses auto-incrementing IDs with indexes on `contentId` (for item-level history lookup), `type` (for filtering by swipe direction), and `timestamp` (for time-range queries and recently shown detection). Dexie.js provides a Promise-based API that is significantly more ergonomic than the native IndexedDB callback interface, and its transaction support ensures data consistency during batch operations like catalog updates. The storage layer also implements a cleanup strategy: old interaction logs (beyond 6 months) are archived to compressed JSON and only summary statistics are retained, preventing unbounded storage growth.

#### 4.4.2. Vector Database: ChromaDB for Prototyping, pgvector for Production

For semantic similarity search and the agentic AI's retrieval pipeline, the system requires a vector database. During development and personal-use prototyping, **ChromaDB** is the recommended choice — it is an open-source, embedded vector database that runs in-process, requires zero infrastructure setup, and can be used directly from JavaScript via its client library. ChromaDB stores the 384-dimensional embeddings with metadata (title, genres, rating) and supports similarity search with metadata filtering (e.g., "find movies similar to X but only in the thriller genre with IMDb rating above 7"). For users who want to scale beyond personal use or enable multi-user cloud features, **pgvector** (the PostgreSQL vector extension) provides a production-grade alternative. Running on managed PostgreSQL (Supabase free tier, 500MB database), pgvector supports HNSW index creation for sub-15ms similarity queries at millions of vectors, full ACID transactions, and the ability to join vector similarity results with relational user data in a single SQL query. The migration path from ChromaDB to pgvector is straightforward: both store the same (vector, metadata) pairs, and only the query syntax changes from ChromaDB's collection API to pgvector's SQL `ORDER BY embedding <=> $query_vector LIMIT 10`. For the personal-use scale of this project (4,300 items), either database provides excellent performance, and the choice primarily depends on whether the user already runs PostgreSQL for other purposes.

The vector database stores two types of embeddings: **item embeddings** (the 384-dimensional vectors precomputed for each catalog item) and **query embeddings** (transient vectors generated from user queries or taste profiles for similarity search). The item embeddings are loaded into the vector database during catalog initialization and updated when the catalog sync pipeline delivers new content. For pgvector, the recommended index is HNSW (Hierarchical Navigable Small World) with `m=16` and `ef_construction=64`, which provides 95%+ recall with sub-10ms query latency for the catalog size. The vector database also supports hybrid search: combining vector similarity with traditional metadata filtering in a single query. For example, "find movies similar to The Dark Knight, but only released after 2010, with IMDb rating above 8.0, not in the blocklist, and not already watched" — this complex query can be expressed as a single SQL statement in pgvector or as a filtered search in ChromaDB, making it ideal for the recommendation pipeline's candidate generation stage.

#### 4.4.3. Redis: Hot Cache for Blocklist and Real-Time Filtering

When the cloud backend is enabled, **Redis** serves as the hot cache for real-time recommendation filtering. The user's blocklist is stored as a Redis SET (`SADD blocklist:user123 contentId`), enabling O(1) membership testing with the `SISMEMBER` command. The watch history is similarly stored as a SET for instant exclusion checks. Recently shown items are tracked in a Redis Sorted Set with TTL-based expiration (`ZADD recently_shown:user123 timestamp contentId` with `EXPIRE` of 48 hours), allowing the recommendation API to efficiently exclude items shown in the current session. The taste vector is cached as a Redis Hash for quick retrieval. These cached data structures enable the `/recommendations` endpoint to execute the full filtering pipeline (blocklist exclusion, watched exclusion, recently shown cooldown, taste vector similarity scoring, and exploration injection) in under 50 milliseconds. Redis also supports Pub/Sub for real-time sync: when the extension adds an item to the blocklist, a Pub/Sub message is broadcast to all connected clients (extension instances on different devices, PWA sessions), ensuring immediate consistency without polling.

Redis's data structure-specific operations are critical for performance. The `SDIFF` command can compute the difference between the candidate set and the blocklist in a single operation, returning only unblocked items. The `ZREVRANGE` command retrieves the highest-scored items from the recommendation ranking. The `ZREMRANGEBYSCORE` command cleans up expired entries from the recently shown set. These atomic operations eliminate the need for application-level iteration and reduce network round-trips. Redis persistence is configured with AOF (Append-Only File) for durability, ensuring that blocklist additions are not lost on restart. For the personal-use scale, Redis can run as a Docker container on the same host as the Preference API, or use a managed service like Upstash (free tier: 10,000 commands/day) for zero-maintenance operation.

### 4.5. External Integrations: APIs and Data Sources

#### 4.5.1. TMDb API: Metadata, Posters, and Watch Providers

The Movie Database (TMDb) API is the primary content metadata source for the recommendation system. It provides comprehensive, community-maintained data on over 800,000 movies and TV shows, with a well-documented REST API and generous free tier. For this project, the critical endpoints are: `GET /search/multi` (search for movies and shows by title, returning TMDb ID, poster, genre IDs, rating, and overview), `GET /movie/{id}` and `GET /tv/{id}` (detailed metadata including cast, crew, runtime, release date, and production companies), `GET /movie/{id}/watch/providers` (streaming availability by region — the `IN` region data includes Jio Hotstar availability), and `GET /genre/movie/list` and `GET /genre/tv/list` (mapping genre IDs to human-readable names). The free tier allows 40 requests every 10 seconds, which is sufficient for weekly catalog enrichment of a few hundred changed titles. TMDb also provides a poster CDN (`https://image.tmdb.org/t/p/w500/{poster_path}`) that can be hot-linked directly in the UI, eliminating the need to host images. The watch providers endpoint is particularly valuable because it offers independent verification of Jio Hotstar availability, complementing the 91mobiles scrape with a second authoritative source.

TMDb API integration requires careful rate limit management. The 40 requests per 10 seconds translates to 4 requests per second, which is easily exceeded during bulk enrichment if not throttled. The catalog sync pipeline implements a token bucket rate limiter that ensures compliance: it maintains a queue of pending API calls and processes them at a maximum rate of 3.5 requests per second, leaving a safety margin. API responses are cached with a 7-day TTL to avoid redundant calls for unchanged titles. The pipeline also handles TMDb's pagination: search results may span multiple pages, and the pipeline follows `page` parameters until all results are retrieved. Error handling includes automatic retry with exponential backoff for 5xx errors, graceful degradation for 404s (title not found in TMDb), and alert logging for 429 rate limit responses (indicating a pipeline bug that needs investigation).

#### 4.5.2. OMDb API: IMDb Ratings and Additional Metadata

The Open Movie Database (OMDb) API serves as the IMDb data source, providing ratings, cast details, plot summaries, and other IMDb-specific fields that TMDb does not include. The primary endpoint is `GET /?i={imdb_id}&apikey={key}` (retrieve full movie details by IMDb ID) or `GET /?t={title}&y={year}&apikey={key}` (search by title and year). The free tier permits 1,000 requests per day, which is adequate for weekly batch enrichment of the Jio Hotstar catalog. OMDb's most valuable field is `imdbRating` (the IMDb score out of 10), which serves as a primary quality signal for the recommendation system. The extension's IMDb rating overlay and the PWA's minimum rating filter both depend on this data. OMDb also provides `Metascore` (Metacritic score), `Ratings` array (including Rotten Tomatoes when available), `Rated` (MPAA/age rating), and `Awards` information. For titles that cannot be matched by IMDb ID (because TMDb does not always include the IMDb ID), the pipeline falls back to title+year search, with fuzzy string matching to handle minor title variations between sources.

OMDb integration complements TMDb rather than replacing it. While TMDb provides richer multimedia (posters, backdrops, videos) and more structured data (genre IDs, production companies), OMDb provides the authoritative IMDb rating that many users trust as a quality signal. The catalog sync pipeline queries both APIs in parallel (within rate limits) and merges the results, preferring TMDb for structured fields and OMDb for IMDb-specific data. The pipeline also detects and handles discrepancies: if TMDb and OMDb report significantly different ratings for the same title, the system logs the discrepancy and uses the more recent rating, or averages them if both are recent. This multi-source approach improves data quality and provides redundancy if one API is temporarily unavailable.

#### 4.5.3. 91mobiles and JustWatch: Catalog Verification

While TMDb and OMDb provide rich metadata, they do not offer a complete, verified list of what is currently available on Jio Hotstar. This gap is filled by two complementary sources. **91mobiles** maintains a dedicated Jio Hotstar movies page that lists 4,317 movies and 1,682 TV shows currently streaming on the platform, updated regularly. The weekly catalog sync pipeline scrapes this page (using BeautifulSoup with respectful delays of 2 seconds between requests) to discover new additions and detect removals. Each entry includes the title, Jio Hotstar URL, age rating, language, genres, cast, and a plot summary, providing a solid baseline for enrichment. **JustWatch** (via its India API or the Parse.bot wrapper) provides structured streaming availability data across all Indian platforms, including Jio Hotstar. The `get_title_details` endpoint returns comprehensive metadata including IMDb, TMDb, and Rotten Tomatoes scores, cast credits, genres, runtime, age rating, and a complete list of streaming offers with provider name, quality tier, and price. The `browse_popular_titles` endpoint provides trending content data. These sources are used for cross-verification: a title is considered "confirmed on Jio Hotstar" only if at least two sources (91mobiles scrape + TMDb watch providers, or 91mobiles + JustWatch) indicate availability. This multi-source verification reduces the risk of recommending content that has been removed from the platform.

The catalog verification pipeline implements a confidence scoring system. A title receives a "high confidence" rating if all three sources (91mobiles, TMDb watch providers, JustWatch) confirm Jio Hotstar availability. It receives "medium confidence" if two sources confirm, and "low confidence" if only one source confirms. Titles with low confidence are still included in the catalog but flagged for manual review or more frequent re-verification. The pipeline also tracks "departing content" — titles that were previously confirmed but are no longer listed by any source — and adds them to a "Leaving Soon" category in the PWA, giving users a chance to watch content before it expires. This feature is particularly valuable in the Indian streaming market where licensing windows are often short and content rotates frequently.

## 5. Key Features and User Experience

### 5.1. Permanent Block and Content Filtering

The cornerstone feature of the recommendation overlay is the ability to permanently remove any movie or show from the user's feed. This is implemented as a multi-level filtering system that operates at different stages of the recommendation pipeline. **Level 1 — Real-Time DOM Filtering (Extension)**: When the user swipes left on a content card or clicks the "Never Show Again" button, the item is immediately added to the blocklist and `display:none` is applied. The MutationObserver continues monitoring for this title (matched by content ID or fuzzy title matching) and hides any future occurrences without requiring a page refresh. This provides instant visual feedback and immediate clutter reduction. **Level 2 — API Filtering (Backend)**: The blocklist is synchronized to the backend (if cloud sync is enabled) and applied as a hard exclusion filter in the `/recommendations` endpoint. Blocked item IDs are excluded via Redis SET difference operations before any scoring or ranking occurs. **Level 3 — Catalog Filtering (Client)**: The PWA and extension both maintain a local copy of the blocklist and apply it client-side as well, ensuring that recommendations work offline and reducing API load. The blocklist supports categorization: users can create custom categories ("Bad Acting", "Wrong Genre", "Already Watched Elsewhere") and apply bulk operations ("Block all movies in this franchise", "Block this director's entire filmography"). A "Hidden Items" page in the PWA allows users to review their blocklist, unblock items they've changed their mind about, and see statistics on their blocking patterns. The system also supports **proactive blocking**: users can search for a title and block it before it ever appears in their recommendations, useful for preemptively filtering upcoming releases from disliked franchises.

The blocklist's effectiveness depends on robust title matching. Jio Hotstar may display the same movie with slightly different titles in different contexts (e.g., "The Dark Knight" vs "Dark Knight" vs "Batman: The Dark Knight"). The matching system uses a three-tier approach: exact content ID match (most reliable, requires the catalog sync pipeline to have mapped Jio Hotstar's internal IDs), fuzzy title match using Levenshtein distance (handles minor variations), and semantic similarity match using embeddings (catches completely different titles for the same movie across languages or re-releases). This multi-tier matching ensures that a blocked item is caught regardless of how Jio Hotstar chooses to display it. The blocklist also propagates across related content: blocking "Avengers: Endgame" can optionally block all other Avengers movies, all Marvel Cinematic Universe content, or all content starring Robert Downey Jr., based on user-configurable propagation rules.

### 5.2. Watched Item Tracking and Re-Recommendation Prevention

Preventing already-watched content from reappearing in recommendations requires a robust tracking system that goes beyond simple binary flags. **Manual Marking**: Users can mark items as watched via the swipe-up gesture or a "Mark as Watched" button. This is the primary input method and gives the user full control. **Automatic Detection (Extension Only)**: The browser extension can infer watch completion by monitoring the user's navigation patterns on Jio Hotstar. If the user visits a movie's watch page and remains there for a duration exceeding 80% of the movie's runtime (obtained from the catalog metadata), the extension automatically marks it as watched. This heuristic is opt-in and requires the user to grant the extension permission to monitor navigation, which is disclosed transparently in the onboarding flow. **Partial Watch Handling**: The system distinguishes between fully watched (100%), mostly watched (50-99%), sampled (10-49%), and abandoned (<10%) content. Fully watched items are completely excluded from recommendations. Mostly watched items are excluded from "new for you" recommendations but may appear in "finish watching" nudges. Sampled items are treated as weak positive signals (the user was interested enough to start) but are not excluded from re-recommendation. Abandoned items are treated as weak negative signals. This nuanced approach prevents the common frustration of being re-recommended a movie that the user watched 90% of and decided not to finish, while still allowing genuinely incomplete content to be surfaced for completion. **Re-watch Allowance**: Users can configure a "re-watch window" (default: 6 months) after which watched items become eligible for re-recommendation. This accommodates users who enjoy rewatching favorite films while preventing immediate re-surfacing.

The watch tracking system also integrates with the "Fresh Feed" guarantee. Even if a user has not explicitly blocked a watched item, it will not appear in the main recommendation feed. Instead, watched items are moved to a separate "Watched" tab in the PWA where users can review their viewing history, rate items they forgot to rate, and see statistics (total watch time, genre distribution, average rating of watched content). This tab becomes a valuable personal viewing diary over time. The system can also detect "watch streaks" — consecutive watches of similar content — and temporarily boost recommendations in that genre, capturing the user's current mood more accurately than a static taste profile.

### 5.3. The Swipe Interface: Left, Right, Up, Down

The swipe interface is the primary interaction model, designed to be intuitive, satisfying, and efficient for rapid content curation. The implementation uses `react-tinder-card` in the PWA and a custom swipe detector in the extension overlay, both providing tactile spring physics powered by Framer Motion. **Swipe Left (Block)**: The card flies off-screen to the left with a red tint and "BLOCKED" stamp animation. The item is permanently added to the blocklist. A brief toast notification appears with an "Undo" button (available for 3 seconds). **Swipe Right (Like)**: The card flies right with a green tint and "LIKED" heart animation. The item's embedding is added to the taste vector with positive weight, and the Two-Tower model is updated. **Swipe Up (Watched)**: The card flies up with a blue tint and checkmark animation. The item is added to watch history with "fully watched" status. **Swipe Down (Watch Later)**: The card flies down with a yellow tint and bookmark animation. The item is added to a "Watch Later" list for future reference. **Tap for Details**: A gentle tap (not swipe) opens a detail modal showing the full metadata panel: IMDb rating, plot, cast, director, runtime, genres, similar titles, and the AI-generated "Why this?" explanation. From the detail view, all four swipe actions are available via button clicks for users who prefer precise control over gestures. The swipe deck intelligently manages its buffer: it preloads the next 10 cards (images and metadata) for smooth transitions and fetches new candidates from the recommendation API when the buffer drops below 5 cards. After every 20 swipes, a brief "breather" card appears showing statistics: "You've liked 12 thrillers and blocked 8 rom-coms. Your top genre this week is Sci-Fi!" These gamification moments make the curation process feel rewarding rather than tedious.

The swipe physics are tuned for satisfaction while preventing accidental actions. The card follows the user's finger with 1:1 tracking during the drag, rotates slightly based on horizontal velocity (simulating physical card flipping), and snaps back to center if released before crossing a 30% threshold of screen width/height. Crossing the threshold triggers the directional action with a satisfying spring animation (stiffness: 300, damping: 20) that provides clear haptic-like feedback even on devices without vibration. The swipe deck also supports keyboard navigation (arrow keys) and accessibility (screen reader announcements for each card's title, rating, and available actions), ensuring the interface is usable by all users regardless of input method or ability.

### 5.4. Fresh Feed on Every Refresh: The Anti-Stagnation Algorithm

The "Fresh Feed" guarantee is achieved through a combination of hard exclusion and intelligent exploration. **Hard Exclusions (100% guaranteed)**: Every recommendation request applies three non-negotiable filters: (1) blocklist exclusion — any blocked item ID is removed before scoring; (2) watched exclusion — any item marked as fully watched is removed (unless the re-watch window has expired); (3) recently shown exclusion — items shown in the last 48 hours are removed via Redis TTL. These exclusions ensure that the user never sees content they have explicitly rejected or already consumed. **Exploration Injection**: After applying hard exclusions, the remaining candidates are scored by the AI layers. However, rather than returning the top-N highest-scored items, the system uses an **epsilon-greedy bandit strategy**: with probability ε (default 0.2, decaying as 1/t where t is session count), the system replaces a recommended item with a random high-quality item from an underexplored genre, a trending new release, or a "serendipity" pick that diverges from the user's typical preferences. This ensures that 20% of recommendations come from outside the user's comfort zone, preventing the filter bubble effect and keeping the feed genuinely fresh. **Session Diversity**: Within a single session, the system tracks which genres, directors, and actors have already been recommended and enforces a diversity constraint: no more than 30% of recommendations may share the same primary genre, no more than 2 from the same director, and no more than 3 starring the same lead actor. **Temporal Freshness**: New releases (added to Jio Hotstar within the last 7 days) receive a temporary boost in their relevance score, ensuring that fresh content is surfaced even if it doesn't perfectly match the user's historical preferences. The combination of these strategies means that even if the user refreshes the page 10 times in a row, each feed will be substantially different from the previous one, with guaranteed novelty and controlled exploration.

The anti-stagnation algorithm also incorporates a "surprise me" mode that users can toggle for maximum diversity. In this mode, epsilon increases to 0.5 (half of recommendations are exploratory), recently shown exclusion extends to 7 days, and the system intentionally surfaces content from genres the user has rarely interacted with. This mode is ideal for users who feel stuck in a rut and want to discover entirely new types of content. Conversely, a "comfort zone" mode reduces epsilon to 0.05, focusing almost entirely on content that closely matches the user's established preferences. These modes give users explicit control over the exploration-exploitation trade-off, something that native platforms never offer.

### 5.5. AI-Explained Recommendations: "Why This?"

One of the most frustrating aspects of native recommendation systems is their opacity — users have no idea why a particular item was suggested. The Agentic AI layer addresses this by generating a natural language explanation for every recommendation. These explanations are produced by the LangGraph reranker (Node 4) and cached to avoid regenerating them for the same recommendation context. Example explanations include: "Recommended because you liked Inception and Interstellar — this is another mind-bending Christopher Nolan film with complex narrative structure" (content similarity + director connection); "You tend to rate crime thrillers above 8/10, and this recently added title has a 8.4 IMDb rating with strong critical acclaim" (genre preference + quality signal); "Since you enjoyed The Social Network and Moneyball, you might appreciate this tech-industry drama based on real events" (thematic similarity across different genres); and "This is outside your usual choices, but it's a highly-rated documentary about a topic you've shown curiosity for in your recent swipes" (exploration justification). Explanations are displayed in the detail modal (tappable "Why this?" button) and as a one-line subtitle under each recommendation card. This transparency builds user trust, helps users understand their own taste patterns, and provides an opportunity for feedback: if an explanation seems wrong ("I didn't actually like Inception that much"), the user can tap "This explanation is wrong" to trigger a taste vector recalibration. The explanation generation uses a carefully crafted prompt template that includes the user's taste profile, the recommended item's metadata, and the specific similarity signals that triggered the recommendation, ensuring that explanations are grounded in actual data rather than generic or hallucinated rationales.

The explanation system also serves an educational purpose. Over time, users begin to understand the patterns in their own preferences: "I apparently really like movies with non-linear narratives" or "I have a strong preference for films under 2 hours." These insights are collected in a "My Taste Insights" section of the PWA, which aggregates patterns detected by the AI across all interactions. This section shows the user's top genres, preferred decades, favorite directors and actors, average runtime preference, and rating distribution — essentially a personal taste dashboard. Users can manually adjust these insights if they feel the AI has mischaracterized them, providing another vector for preference refinement. The combination of per-recommendation explanations and aggregate taste insights creates a feedback loop where the user understands the AI, the AI understands the user, and both improve together.

## 7. Legal, Ethical, and Technical Safety Considerations

### 7.1. Terms of Service Analysis

The proposed architecture is designed to operate within the boundaries of Jio Hotstar's Terms of Service and established legal precedents for user-controlled browser modifications. The critical distinction is between **content access** (which the architecture never attempts to bypass) and **content presentation** (which the architecture modifies only on the user's own device). Jio Hotstar's ToS prohibits unauthorized access to their systems, reverse engineering of their applications, circumvention of technical protections, and redistribution of content. The browser extension does none of these things: it operates after content has been legitimately delivered to an authenticated user's browser, it does not scrape backend APIs or harvest credentials, it does not proxy or decrypt video streams, and it does not redistribute content or metadata to third parties. The standalone PWA is entirely independent, sourcing its own metadata from public APIs and linking to Jio Hotstar only via standard HTTP deep links. This approach falls into the same category as ad blockers, Stylus (custom CSS), and accessibility tools — all of which modify how content is presented in the user's browser without violating platform ToS. No court has ruled against personal-use browser extensions that alter DOM presentation, and the established legal interpretation is that users have the right to control their own browsing experience.

The Terms of Service analysis must also consider the specific language that Jio Hotstar uses. While the exact ToS text evolves, streaming platforms typically prohibit: (a) using automated tools to access their services, (b) modifying or creating derivative works from their content, (c) interfering with other users' access, and (d) using their trademarks without permission. The proposed architecture does not violate any of these: (a) the extension does not automate access — it observes content that the user has already legitimately loaded; (b) it does not modify the content itself, only its presentation in the user's browser; (c) it does not affect any other user; and (d) it does not use Jio Hotstar trademarks in commerce. The deep linking to Jio Hotstar URLs is explicitly permitted — platforms generally encourage deep linking because it drives traffic and engagement.

### 7.2. DRM and Security Boundary Respect

The architecture strictly respects Jio Hotstar's Digital Rights Management (DRM) and security perimeter. Video content on Jio Hotstar is protected by multi-DRM: Google Widevine for Android and web browsers, Apple FairPlay for iOS, and Microsoft PlayReady for Windows. These DRM systems encrypt video streams and require hardware-backed decryption using keys that are never exposed to application code. The architecture does not attempt to intercept video streams, extract decryption keys, proxy network traffic, or embed video content in unauthorized contexts. All playback is delegated to Jio Hotstar's native application or website via deep links, where the standard DRM authentication and decryption chain operates unchanged. The extension only manipulates the HTML/CSS presentation layer (text, images, layout) and never touches the video player or streaming components. This zero-tolerance approach to DRM ensures complete technical and legal safety, eliminating any risk of copyright infringement or circumvention liability. The deep link engine generates only navigation URLs — it does not attempt to embed, frame, or proxy the video player.

The DRM boundary is non-negotiable. Even if technical workarounds were possible (and they are increasingly difficult with modern DRM), attempting them would expose the user to severe legal liability under the Digital Millennium Copyright Act (DMCA) and similar legislation worldwide. The deep-link-only approach to playback is the only viable strategy, and it is fully sufficient: users click "Watch" and are taken to the official Jio Hotstar app or website where the content plays normally with all DRM protections intact. The overlay system's value is in the discovery and curation phase that precedes playback, not in the playback itself.

### 7.3. Privacy: Local-First Data Model

Privacy is a core architectural principle. All sensitive user data — blocklists, watch history, taste vectors, swipe logs, and AI-generated taste profiles — is stored locally on the user's device by default. `chrome.storage.local` and IndexedDB provide persistent, private storage that never leaves the device unless the user explicitly enables cloud sync. The local-first model means: no user account is required; no personal data is transmitted to external servers; the user's viewing preferences cannot be accessed by the developers, advertisers, or third parties; and the system works fully offline after the initial catalog download. When cloud sync is enabled (opt-in), data is transmitted over HTTPS to the user's self-hosted backend or a trusted service, with AES-256 encryption at rest. The local LLM execution via Ollama ensures that taste profiles and conversation history never reach cloud AI providers. This privacy model stands in stark contrast to native streaming platforms, which extensively mine user viewing data for advertising and content licensing decisions.

The local-first model also provides practical benefits beyond privacy. It eliminates the need for user authentication (reducing friction), ensures the system works offline (improving reliability), and removes the backend as a single point of failure (improving availability). The only data that leaves the device by default is API calls to TMDb and OMDb — and these contain only movie titles and IDs, never user identifiers or viewing history. Even these API calls can be proxied through a privacy-preserving middleware if desired. The architecture treats user data as the user's property, not as a monetizable asset.

### 7.4. Rate Limiting and API Etiquette

All external API integrations are designed to operate within free tier limits and follow responsible usage practices. The TMDb API free tier (40 requests per 10 seconds) is respected through client-side request batching and server-side rate limiting using a token bucket algorithm. The OMDb API free tier (1,000 requests per day) is managed by caching responses for 24 hours and only fetching data for titles that the user actually encounters. The 91mobiles scraper implements polite crawling: 2-second delays between requests, respect for `robots.txt`, and requests only during off-peak hours. The backend API implements rate limiting (100 requests/hour per device) to prevent abuse. All API calls include proper User-Agent headers identifying the application and a contact email, following API provider guidelines. These practices ensure sustainable, long-term operation without incurring costs or risking API access revocation.

The rate limiting strategy also includes graceful degradation. If an API rate limit is temporarily exceeded, the system falls back to cached data and queues the failed request for retry. If an API is unavailable for an extended period, the system operates entirely on its local catalog cache (which may be slightly stale but still functional). The catalog sync pipeline also implements a "gentle" mode that reduces request frequency when API quota is running low, ensuring that critical functions (blocklist sync, recommendation scoring) always have sufficient quota while non-critical enrichment (updating cast photos, refreshing trending data) can be deferred.

## 8. Conclusion: A New Paradigm for Personal Content Curation

### 8.1. What This Architecture Achieves

This hybrid architecture delivers something that no existing streaming platform provides: **genuine user control over content discovery**. It transforms the passive consumption experience into an active curation relationship, where the user's explicit preferences — communicated through intuitive swipe gestures — continuously shape a personalized content universe. The system eliminates the three primary frustrations of native recommendation: unwanted content clutter (via permanent blocking), repetitive re-recommendation (via watched-item tracking), and algorithmic stagnation (via the Fresh Feed guarantee with exploration injection). It adds capabilities that native platforms lack: natural language search for content discovery, AI-generated explanations for every recommendation, a transparent view of the user's own taste profile, and cross-platform unification. Most importantly, it achieves all of this without compromising security, violating Terms of Service, or sacrificing privacy. The local-first data model ensures that personal viewing preferences remain personal. The deep-link-only playback strategy ensures full DRM compliance. The browser extension operates within established legal boundaries for user-controlled client modifications. The standalone PWA provides a fully independent alternative that requires no interaction with Jio Hotstar's systems at all. This architecture demonstrates that it is possible to build a genuinely better recommendation experience without fighting the platform — by operating alongside it, augmenting it, and ultimately giving the user the control they have always deserved.

The architecture also achieves something subtler but equally important: it makes the recommendation process transparent and educational. Native platforms treat their algorithms as black boxes, leaving users frustrated when recommendations seem random or inappropriate. This system exposes the reasoning behind every recommendation, shows users patterns in their own preferences, and invites them to actively refine their taste profile. Over time, users develop a better understanding of their own taste, which makes them more satisfied consumers of content — not just on Jio Hotstar, but across all their streaming services. The system doesn't just recommend movies; it helps users understand why they like what they like.

### 8.2. Future Directions

The architecture is designed for extensibility. Beyond Phase 6's multi-platform expansion, several future directions are naturally supported. **Social Curation**: Users could create and share blocklists ("Block all Marvel movies"), taste profiles ("Import my friend's sci-fi preferences"), and curated playlists ("Best Indian thrillers on Jio Hotstar"). **Advanced AI**: The LangGraph pipeline can be extended with additional nodes for mood detection ("you seem to browse more comedies on weekends"), temporal preference modeling (tastes that shift by season or time of day), and cross-media recommendations ("you liked this book, here's the movie adaptation"). **Content Creation Integration**: The system could integrate with review platforms (Letterboxd, IMDb reviews) to factor community sentiment into recommendations, or with calendar apps to suggest content based on available watch time. **Voice Interface**: The natural language search can be extended to full voice control ("show me something like Inception but shorter and funnier"). **Predictive Preloading**: Based on the user's typical watch times, the system could preload catalog metadata in advance, ensuring that the swipe deck is always ready with fresh recommendations at the moment the user opens the app. Each of these extensions builds on the existing architecture without requiring fundamental redesign.

The most exciting future direction is the potential for this system to become a universal content curation layer across all media types. The same architecture — blocklist, taste vector, semantic embeddings, agentic AI — could be applied to music (Spotify, Apple Music), podcasts, news articles, books, and even shopping recommendations. The user's taste profile, refined through thousands of interactions across media types, becomes a genuinely portable representation of their preferences — one that they own, control, and can apply to any discovery context. This is the long-term vision: not just a better way to find movies on Jio Hotstar, but a fundamental rethinking of how algorithmic recommendations work — putting the user, not the platform, in control.

### 8.3. The Bigger Picture: User Control in the Algorithmic Age

This project, while technically focused on Jio Hotstar movie recommendations, represents a broader principle: **users should have meaningful control over the algorithmic systems that shape their information and entertainment consumption**. The current paradigm, where platforms optimize for engagement metrics with minimal user override, creates filter bubbles, recommendation fatigue, and a sense of algorithmic helplessness. The architecture proposed here demonstrates an alternative: algorithms that learn from explicit, intentional user feedback rather than passive behavioral inference; systems that explain their reasoning rather than operating as black boxes; and tools that prioritize individual preference over aggregate engagement. As algorithmic curation becomes increasingly central to how we discover news, entertainment, products, and even relationships, the principles embedded in this architecture — local-first privacy, user-controlled filtering, explainable AI, and open data portability — become not just technical preferences but essential safeguards for individual autonomy in the digital age. Building a better movie recommender is a worthwhile goal in itself. But demonstrating that users can take back control of their algorithmic experiences — legally, ethically, and effectively — is the project's most significant contribution.

The bigger picture also includes a call to action for the tech industry. Streaming platforms could voluntarily provide the APIs and hooks that would make third-party curation tools easier to build and more reliable. They could expose blocklist APIs, recommendation explanation endpoints, and standardized metadata formats. They could adopt the Principles of User Control in Algorithmic Systems: explicit feedback mechanisms, transparent recommendation logic, portable preference profiles, and respect for user-defined boundaries. The architecture in this report works around the absence of these APIs, but it would work far better — and benefit far more users — if platforms embraced rather than resisted user-controlled curation. The future of streaming discovery is not better black-box algorithms; it is algorithms that work transparently in service of the user's explicitly stated preferences. This project is a step toward that future.
